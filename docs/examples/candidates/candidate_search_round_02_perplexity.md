**Candidate Search Round 01**

## Current state

The strongest current state is that this search surface is not broad and ambiguous anymore. It has already collapsed into a small set of high-value prime pipelines where a reject-only DCI/GWR step is structurally plausible, benchmarkable, and easy to explain to outsiders. The most important non-obvious point is that the best targets are not generic number theory libraries just because they touch primes. The best targets are software systems where prime handling sits in a repeated candidate-survival loop with a cheap front gate and an expensive back gate. That architecture matters more than whether the project is “about primes.” In practical terms, the best candidates are the ones where the method can sit between trial division or small-prime screening and Miller-Rabin or BPSW, reducing expensive confirmations without changing correctness guarantees. That is why cryptographic key generation libraries and large-integer `nextprime` paths dominate the ranking, while pure sieving projects score lower even if they are prime-heavy. [github](https://github.com/Legrandin/pycryptodome/blob/master/lib/Crypto/Math/Primality.py)

A second non-obvious point is that the best first target is not necessarily the most important software in the world, but the place where the method can be demonstrated cleanly with the least friction. That is why PyCryptodome stands above OpenSSL even though OpenSSL has larger downstream deployment. PyCryptodome offers nearly the same structural story, but with a much easier proof path, a smaller language interface, and a direct smoke-test path inside the current repo. SymPy lands just behind it because its `nextprime` and `isprime` structure gives a nearly textbook insertion point with strong correctness checks. OpenSSL remains strategically important, but its higher review bar and C integration cost push it slightly down despite the very large practical upside. In other words, the current state of the search is favorable because the “best first patch,” the “best first smoke test,” and the “best public benchmark story” are mostly aligned rather than pulling in different directions. [github](https://github.com/sympy/sympy/blob/master/sympy/ntheory/generate.py)

## Shortlist

The top 3 candidates to evaluate first are PyCryptodome, SymPy, and OpenSSL. PyCryptodome is the best immediate next step because the current repo already documents a concrete CDL cryptographic prefilter path and benchmark results, which means the shortest path to a repeatable smoke test is already available. SymPy is the best adjacent target because its `nextprime` loop offers an equally clear Python insertion point with strong correctness infrastructure. OpenSSL is the best strategic stretch target because its deployment footprint makes any measurable improvement unusually important, even though integration is harder. [github](https://github.com/Legrandin/pycryptodome/blob/master/lib/Crypto/Math/Primality.py)

The single best first smoke-test target is PyCryptodome. It combines a narrow insertion point, direct benchmark relevance, existing repo evidence, and the clearest “real improvement” story for outsiders. [github](https://github.com/Legrandin/pycryptodome/blob/master/lib/Crypto/Util/number.py)
